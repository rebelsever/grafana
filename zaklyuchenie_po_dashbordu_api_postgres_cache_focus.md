# Заключение по результатам анализа дэшборда API ↔ PostgreSQL за последние 24 часа

## 1. Цель заключения

Цель анализа — определить, какие endpoint’ы имеет смысл рассматривать для кеширования, где кеширование нецелесообразно, а где основной эффект даст оптимизация SQL-запросов и прикладной логики.

Документ ориентирован на принятие практического решения, а не на детальное техническое расследование.

---

## 2. Ключевые наблюдения по системе

По данным дэшборда за последние 24 часа:

- **RPS API:** ~1.26k req/s
- **p95 latency API:** ~1.748 s
- **Утилизация пула соединений:** ~1.23%
- **Ожидающие соединения:** 0
- **Backend-сессии PostgreSQL:** ~2k
- **PostgreSQL cache hit ratio:** ~96.03%

### Вывод по инфраструктурному состоянию
На текущих данных **не подтверждается гипотеза**, что основная проблема связана:
- с нехваткой connection pool;
- с общей аварийной деградацией PostgreSQL;
- с накоплением dead tuples;
- с выраженной lock-контенцией.

Следовательно, основной источник деградации — **не “состояние БД в целом”, а отдельные тяжелые endpoint’ы и их запросная/прикладная логика**.

---

## 3. Методика оценки потенциального эффекта

Для оценки приоритета использована уже заложенная в дэшборд метрика нагрузки:

`Score = Traffic × Avg Latency`

Эта величина не является прямым DB time, но подходит как практический прокси-показатель того, **какой endpoint дает наибольший совокупный эффект на путь API → приложение → БД**.

### 3.1. Как считался потенциальный эффект от кеширования
Теоретический эффект оценивался по формуле:

`Potential Cache Benefit = Score × Hit Ratio × Bypass Ratio`

Где:
- **Score** — значение из панели дэшборда;
- **Hit Ratio** — ожидаемая доля попаданий в кеш;
- **Bypass Ratio** — доля backend-нагрузки, которая реально исчезает при cache hit.

Для **основных кандидатов на response cache** принят базовый расчетный сценарий:

- **Hit Ratio = 50%**
- **Bypass Ratio = 70%**

Тогда:

`Potential Cache Benefit ≈ Score × 0.35`

То есть теоретически можно ожидать около **35% снижения совокупной нагрузки по конкретному endpoint**.

### 3.2. Как считался потенциальный эффект от SQL-оптимизации
Для методов, где кеш не выглядит основной мерой, эффект оценивался как снижение текущей стоимости endpoint’а:

`Potential SQL Benefit = Score × Optimization Effect`

Для базового сценария принято:
- **Optimization Effect = 30%**

То есть:

`Potential SQL Benefit ≈ Score × 0.30`

Это не обещание результата, а разумная оценка для выбора приоритета работ.

---

## 4. Общий вывод по кешированию

Кеширование **имеет смысл рассматривать выборочно**, в первую очередь для read-endpoint’ов без высокой кардинальности ключа и без явной зависимости от быстро меняющегося path-параметра.

При этом в рамках данного заключения принимается практическое допущение:

**endpoint’ы с переменной в конце URI не рассматриваются как приоритетные кандидаты на full response cache**, так как для них ожидается высокий разброс ключей и низкий hit ratio.

Такие методы могут оставаться кандидатами только на:
- fragment cache;
- cache reference data;
- cache вычисленных промежуточных результатов;
- hot-key cache при подтвержденной повторяемости запросов.

---

## 5. Что имеет смысл рассмотреть для кеширования

## 5.1. Основные кандидаты

### 1. `GET /v2/user_roles/permissions`
**Score:** 3.29  
**Почему кандидат:**
- read-only endpoint;
- нет переменной в конце URI;
- результат логически похож на права/разрешения, которые обычно меняются значительно реже, чем читаются.

**Теоретический эффект:**
- `3.29 × 0.35 = 1.15`

**Вывод:**
Хороший кандидат на **response cache** или короткий Redis cache с контролируемым TTL / инвалидацией по изменению ролей.

---

### 2. `GET /v1/contexts/project`
**Score:** 1.54  
**Почему кандидат:**
- read-only;
- стабильный URI;
- контекст проекта, как правило, читается чаще, чем меняется.

**Теоретический эффект:**
- `1.54 × 0.35 = 0.54`

**Вывод:**
Хороший кандидат на кеширование. Вероятный эффект — снижение нагрузки на чтение из БД и уменьшение latency.

---

### 3. `GET /v3/projects`
**Score:** 0.88  
**Почему кандидат:**
- read-only;
- стабильный endpoint;
- по смыслу похож на справочный/списочный запрос.

**Теоретический эффект:**
- `0.88 × 0.35 = 0.31`

**Вывод:**
Кандидат второго приоритета. Сам по себе не даст максимальный эффект, но может быть хорошей “быстрой победой”.

---

### 4. `GET /rest/api/permission/v1/space`
**Score:** 0.403  
**Почему кандидат:**
- read-only;
- без высокой кардинальности ключа;
- вероятно, данные часто повторяются между запросами.

**Теоретический эффект:**
- `0.403 × 0.35 = 0.14`

**Вывод:**
Небольшой, но понятный и безопасный кандидат на кеширование.

---

### 5. `GET /v4/organizations/{orgKey}/projects`
**Score:** 0.65  
**Почему кандидат:**
- формально содержит переменную;
- однако переменная не выглядит как высокоизменяемый технический идентификатор наподобие issueId;
- если число `orgKey` ограничено, а обращения повторяются, возможен приемлемый hit ratio.

**Теоретический эффект (только при подтверждении повторяемости):**
- `0.65 × 0.25 = 0.16`  
  Здесь принят более осторожный коэффициент, чем для основных кандидатов.

**Вывод:**
Не базовый, а **условный кандидат**. Рассматривать только после проверки повторяемости ключей.

---

## 5.2. Суммарный эффект по основным кандидатам на full response cache

Берем 4 основных кандидата без высококардинальных path-параметров:

- `/v2/user_roles/permissions` → 3.29
- `/v1/contexts/project` → 1.54
- `/v3/projects` → 0.88
- `/rest/api/permission/v1/space` → 0.403

**Суммарный Score:**  
`3.29 + 1.54 + 0.88 + 0.403 = 6.113`

**Теоретический профит при базовом сценарии:**  
`6.113 × 0.35 = 2.14`

### Практический вывод
Если эти методы действительно повторяемы и допускают TTL/инвалидацию, то кеширование **может дать ощутимое снижение читающей нагрузки на БД и заметный локальный эффект на latency**, при этом риск реализации ниже, чем у методов с высококардинальными path-параметрами.

## 6. Что кешировать нецелесообразно или малореалистично

## 6.1. Не рассматривать как приоритетные кандидаты на full response cache

### `GET /rest/api/unit/v2/{code}`
**Score:** 9.77  
**Почему не брать в первый приоритет на full response cache:**
- ключ содержит переменный параметр `{code}`;
- ожидается высокая кардинальность;
- без доказанной повторяемости запросов hit ratio может оказаться низким.

**Вывод:**
Несмотря на то, что endpoint является самым “дорогим” по рейтингу, **как первый кандидат на обычный response cache он рискован**.  
Если брать его в работу — то только после проверки:
- повторяемости `code`;
- количества hot keys;
- реальной локальности запросов.

Возможный путь — не full response cache, а **fragment cache** или кеширование отдельных reference-данных внутри метода.

---

### `GET /extension/plugin/v2/rest/api/jira/rest/api/2/issue/{issueIdOrKey}`
**Score:** 4.61  
**Почему не брать в первый приоритет:**
- переменный идентификатор в конце URI;
- вероятна зависимость от интеграции и внешнего источника;
- без понимания повторяемости ключей и характера backend-call кеширование может не дать ожидаемого эффекта.

**Вывод:**
Не первоочередной кандидат на Redis response cache.

---

### `GET /rest/api/unit/files/v1/download`
**Score:** 1.80  
**Почему не брать в первый приоритет:**
- download endpoint не всегда рационально кешировать целиком;
- эффект может упираться не в БД, а в файл/хранилище/сеть;
- нужно отдельно проверять payload и сценарий использования.

**Вывод:**
Не рекомендован как быстрый кандидат на full response cache.

---

### `GET /rest/api/permission/v1/space/{unitCode}`
**Score:** 0.53  
### `GET /extension/plugin/v2/rest/api/swtr_wiki_plugin/v2/wiki/unit/{unitCode}`
**Score:** 0.687  

**Почему не брать в первый приоритет:**
- переменная в URI;
- ожидаемо ниже вероятность повторяемости;
- умеренный интегральный Score.

**Вывод:**
Кандидаты только на вторую очередь и только после проверки key locality.

---

## 6.2. Что точно не следует решать кешем

### `POST /public/v1/releases`
### `POST /api/v1/productVersions/{productVersionEntityId}/planCosts`
### `POST /v1/worklog`
### `PUT /v4/organizations/{org_key}/projects/{project_key}/users/{user_id}/roles`
### `DELETE /v4/organizations/{org_key}/projects/{project_key}/users/{user_id}/roles`

**Причина:**
- это write-методы;
- проблема лежит не в повторяемом чтении, а в самой стоимости операции;
- кеш не устраняет дорогую запись, транзакции, блокировки и неэффективные SQL.

**Вывод:**
Для этих методов правильный вектор — **SQL/ORM/логическая оптимизация**, а не Redis.

## 8. Рекомендуемое управленческое решение

## 8.1. Что брать в работу на первую волну

### Направление 1. Точечное кеширование read-endpoint’ов
Взять в проработку:
- `GET /v2/user_roles/permissions`
- `GET /v1/contexts/project`
- `GET /v3/projects`
- `GET /rest/api/permission/v1/space`

**Ожидаемый результат:**  
контролируемое снижение read-нагрузки на БД и локальное улучшение latency при относительно низком риске.

---

### Направление 2. SQL/логическая оптимизация тяжелых write/service endpoint’ов
Взять в проработку:
- `POST /public/v1/releases`
- `POST /search`
- `POST /extension/.../hierarchy`
- `POST /api/v1/productVersions/.../planCosts`
- `POST /v1/worklog`
- `PUT/DELETE ... /roles`

**Ожидаемый результат:**  
снижение стоимости тяжелых операций, улучшение latency, уменьшение локальной нагрузки на БД и внутренних backend-процессов.

---

## 8.2. Что не брать в первую волну кеширования

Не брать как первоочередные кандидаты на full response cache:
- `GET /rest/api/unit/v2/{code}`
- `GET .../issue/{issueIdOrKey}`
- `GET .../space/{unitCode}`
- `GET .../wiki/unit/{unitCode}`
- `GET .../download`

**Причина:** высокая вероятность низкого hit ratio из-за кардинальности ключей или неподходящей природы endpoint’а.

---

## 9. Финальный вывод

По данным дэшборда за последние 24 часа наиболее рациональная стратегия состоит не в массовом введении кеша, а в **комбинированном подходе**:

1. **Точечно кешировать стабильные read-endpoint’ы** без высокой кардинальности ключа.  
   Это может дать ограниченный, но управляемый и быстрый эффект.

2. **Основной технический резерв искать в SQL и прикладной логике тяжелых write/service-методов**.  
   По текущим данным именно это направление обещает более высокий суммарный эффект.

### Короткий вывод для принятия решения
- **Да, кеширование имеет смысл**, но **только для ограниченного набора стабильных read-endpoint’ов**.
- **Нет, кеширование не является универсальным решением** для основных проблемных методов текущего периода.
- **Наибольший потенциальный эффект** в текущей картине, вероятнее всего, даст **оптимизация тяжелых SQL/транзакционных сценариев**, а не первая волна Redis сама по себе.
