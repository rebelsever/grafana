# Заключение по результатам анализа дэшборда API ↔ PostgreSQL за последние 24 часа

## 1. Краткое описание поведения системы

| Показатель | Значение | Интерпретация |
|---|---:|---|
| RPS API | ~1.26k req/s | Нагрузка на API стабильная и заметная |
| p95 latency API | ~1.748 s | Подтверждается деградация по времени ответа |
| Утилизация пула соединений | ~1.23% | Признаков saturation пула нет |
| Ожидающие соединения | 0 | Очереди на connection pool нет |
| Backend-сессии PostgreSQL | ~2k | Нагрузка на БД есть, но без явного кризиса по подключениям |
| PostgreSQL cache hit ratio | ~96.03% | Хороший показатель чтения из памяти |

### Общий вывод по состоянию системы

На текущих данных не подтверждается, что основная проблема вызвана:
- нехваткой connection pool;
- общей деградацией PostgreSQL как системы;
- bloat / dead tuples;
- массовой lock-контенцией.

Основной вывод: проблема объясняется не общим состоянием БД, а набором отдельных тяжелых endpoint’ов и их запросной/прикладной логикой.

---

## 2. Методика оценки потенциального эффекта

Для оценки приоритета использован интегральный показатель:

`Score = Traffic × Avg Latency`

Он не равен прямому DB time, но подходит как практический показатель совокупной нагрузки на путь API → приложение → БД.

| Направление | Формула | Принятый коэффициент |
|---|---|---:|
| Кеширование | `Потенциальный эффект = Score × 0.35` | 50% hit ratio × 70% backend bypass |
| SQL-оптимизация | `Потенциальный эффект = Score × 0.30` | 30% снижение стоимости endpoint’а |

> Это расчетная оценка для выбора приоритета, а не гарантия фактического эффекта.

## 3. Endpoint’ы, которые имеет смысл рассмотреть для кеширования

В таблицу включены endpoint’ы, которые:
- являются read-only;
- не имеют явно высококардинального переменного ключа в конце URI;
- логически похожи на справочные / повторяемые чтения;
- выглядят подходящими для Redis response cache в первой волне.

| Endpoint | Основание для рассмотрения | Ограничения | Приоритет |
|---|---|---|---|
| `GET /v2/user_roles/permissions` | Read-only, стабильный URI, вероятно высокая повторяемость | Нужна проверка частоты изменений прав | Высокий |
| `GET /v1/contexts/project` | Read-only, стабильный URI, вероятно повторяемый контекст | Нужна проверка актуальности данных | Высокий |
| `GET /v3/projects` | Read-only, стабильный URI, похож на справочный список | Эффект ниже, чем у лидеров | Средний |
| `GET /rest/api/permission/v1/space` | Read-only, стабильный endpoint, умеренный риск | Нужна проверка бизнес-критичности актуальности | Средний |

### Совокупный теоретический эффект по первой волне кеширования

| Показатель | Значение |
|---|---:|
| Суммарный Score группы | 6.113 |
| Расчетный коэффициент эффекта | 0.35 |
| Теоретический суммарный эффект | **2.14** |

### Вывод по направлению кеширования

Точечное кеширование перечисленных endpoint’ов выглядит оправданным. Это не универсальное решение для всей системы, но может дать локально заметное снижение нагрузки на БД и улучшение latency при сравнительно низком риске реализации.

## 4. Endpoint’ы, которые нецелесообразно брать в первую волну кеширования

### 4.1. Read-endpoint’ы с высокой кардинальностью ключа или спорной природой ответа

| Endpoint | Почему нецелесообразно брать в первую волну кеширования | Что можно рассмотреть вместо этого |
|---|---|---|
| `GET /rest/api/unit/v2/{code}` | Переменная в конце URI, вероятно высокая кардинальность ключей | Проверка hot keys, fragment cache, кеш reference-данных |
| `GET /extension/plugin/v2/rest/api/jira/rest/api/2/issue/{issueIdOrKey}` | Переменный ключ, возможна зависимость от внешней интеграции | Локальный cache hot keys, оптимизация интеграционного сценария |
| `GET /rest/api/unit/files/v1/download` | Download endpoint, неочевидная выгода от full response cache | Проверка bottleneck: БД, файловое хранилище, сеть |
| `GET /rest/api/permission/v1/space/{unitCode}` | Переменная в конце URI, ожидаемо ниже hit ratio | Анализ повторяемости `unitCode` |
| `GET /extension/plugin/v2/rest/api/swtr_wiki_plugin/v2/wiki/unit/{unitCode}` | Переменная в конце URI, спорная повторяемость | Анализ реальной key locality |

### 4.2. Write-endpoint’ы, для которых кеш не является решением

| Endpoint | Почему кеширование нецелесообразно |
|---|---|
| `POST /public/v1/releases` | Write-операция, проблема в стоимости самой бизнес-операции |
| `POST /api/v1/productVersions/{productVersionEntityId}/planCosts` | Write-операция, требуется оптимизация обработки / SQL |
| `POST /v1/worklog` | Write-операция, кеш не снижает стоимость записи |
| `PUT /v4/organizations/{org_key}/projects/{project_key}/users/{user_id}/roles` | Изменяющий сценарий, нужен разбор транзакций и SQL |
| `DELETE /v4/organizations/{org_key}/projects/{project_key}/users/{user_id}/roles` | Изменяющий сценарий, кеш не решает корневую причину |
| `POST /error` | Технический endpoint, не является продуктовым кандидатом на кеширование |
| `POST UNKNOWN` | Техническая / неидентифицированная запись, не кандидат на кеш |

### Вывод по неподходящим для кеширования endpoint’ам

Основная причина отказа от кеширования в первой волне — либо высокая кардинальность ключа, либо write-семантика. Для этих методов ожидаемый эффект от Redis либо сомнителен, либо существенно ниже эффекта от оптимизации бизнес-логики и SQL.

## 5. Endpoint’ы, для которых основным решением должна стать SQL-оптимизация

| Endpoint | Основание для оптимизации | Предполагаемое направление работ | Приоритет |
|---|---|---|---|
| `POST /public/v1/releases` | Самый проблемный write-метод, p95 = 11.1 s | SQL, транзакции, бизнес-логика | Высокий |
| `POST /search` | Высокий интегральный Score, вероятно тяжелые запросы/сортировки | SQL, индексы, сокращение стоимости запроса | Высокий |
| `POST /extension/plugin/v2/rest/api/swtr_wiki_plugin/v1/wiki/unit/hierarchy` | Высокий интегральный Score | SQL, запросная модель, возможно batching | Высокий |
| `POST /api/v1/productVersions/{productVersionEntityId}/planCosts` | Очень высокий p95 | SQL, логика расчета, разбор количества запросов | Высокий |
| `POST /v1/worklog` | Заметимая latency у write-операции | SQL, каскадные обновления, транзакция | Средний |
| `PUT /v4/organizations/{org_key}/projects/{project_key}/users/{user_id}/roles` | Высокая средняя latency write-операции | SQL, транзакции, число обращений к БД | Средний |
| `DELETE /v4/organizations/{org_key}/projects/{project_key}/users/{user_id}/roles` | Высокая средняя latency write-операции | SQL, транзакции, удаление зависимостей | Средний |

### Совокупный теоретический эффект по первой волне SQL-оптимизации

В расчет включены endpoint’ы с наиболее выраженным интегральным Score:
- `POST /public/v1/releases`
- `POST /search`
- `POST /extension/.../hierarchy`

| Показатель | Значение |
|---|---:|
| Суммарный Score группы | 8.85 |
| Расчетный коэффициент эффекта | 0.30 |
| Теоретический суммарный эффект | **2.66** |

> Методы, попавшие в список по p95/avg latency без достаточного интегрального Score, оставлены в таблице как кандидаты на анализ, но не включены в суммарный расчет.

### Вывод по направлению SQL-оптимизации

На текущих данных именно SQL/ORM/логическая оптимизация тяжелых write/service endpoint’ов выглядит направлением с наибольшим ожидаемым эффектом. По теоретической оценке первая волна SQL-оптимизации может дать больший суммарный эффект, чем первая волна кеширования.

## 6. Результаты анализа DBA-метрик

| Наблюдение | Вывод |
|---|---|
| Cache hit ratio PostgreSQL ~96.03% | БД хорошо обслуживает чтение из памяти |
| Pending connections = 0, pool utilization ~1.23% | Нет признаков saturation connection pool |
| Dead tuples по таблицам фактически отсутствуют | Bloat не является значимой причиной деградации |
| Массовой lock-контенции не наблюдается | Блокировки не выглядят основной причиной проблемы |
| В топах seq scan / index ratio преобладают служебные, системные и audit-таблицы | DBA-топы не указывают напрямую на продуктовую корневую причину |

### Общий вывод по DBA-части

Текущие метрики БД не подтверждают, что корневая проблема связана с общим “плохим состоянием PostgreSQL”. Наиболее вероятно, что деградация вызвана локально тяжелыми endpoint’ами и их прикладной логикой.

## 7. Итоговое управленческое решение

| Направление | Решение | Ожидаемый эффект |
|---|---|---|
| Первая волна кеширования | Запускать точечно для стабильных read-endpoint’ов без высокой кардинальности ключа | Умеренный, но быстрый и управляемый |
| Массовое кеширование read-endpoint’ов с переменной в конце URI | Не запускать без проверки повторяемости ключей | Высокий риск низкого hit ratio |
| Оптимизация тяжелых write/service endpoint’ов | Запускать в первую очередь | Наибольший ожидаемый суммарный эффект |
| Массовые действия на уровне БД (vacuum/bloat/locks) | Не рассматривать как главный приоритет | Метрики не подтверждают системную проблему БД |

---

## 8. Финальный вывод

По результатам анализа за последние 24 часа кеширование имеет смысл рассматривать **только выборочно**, в первую очередь для стабильных read-endpoint’ов без высокой кардинальности ключа. Наиболее реалистичными кандидатами первой волны являются `GET /v2/user_roles/permissions`, `GET /v1/contexts/project`, `GET /v3/projects` и `GET /rest/api/permission/v1/space`.

При этом основная техническая выгода на текущих данных ожидается не от массового Redis-кеширования, а от **оптимизации SQL и прикладной логики тяжелых write/service endpoint’ов**, прежде всего `POST /public/v1/releases`, `POST /search` и `POST /extension/.../hierarchy`.

Таким образом, рациональная стратегия — комбинированная:
1. точечное кеширование ограниченного списка стабильных read-endpoint’ов;
2. приоритетная оптимизация тяжелых write/service-сценариев;
3. отказ от первой волны кеширования для endpoint’ов с высококардинальными path-параметрами без подтверждения реальной повторяемости ключей.
